3. §7の構造規約（見出し番号・問題ブロック書式・用語フィールド名）を崩さない。**特に「解説:」は1行に収める**（複数行にするとパースが途切れる）。

5. 問題・本文はすべてオリジナルを維持。公式問題・推薦図書本文の複製・翻案をしない。
6. 本文書中の行番号は 2026-07-02 時点のもの。**編集が進むと行番号はずれるため、必ず併記のアンカー（問番号・見出し・用語名・引用文字列）で grep して位置を特定してから編集する。**

## 1. 現状評価の要約

### 1.1 維持すべき強み（壊さないこと）

1. **事実の正確性は高水準**。ISO 9241-210/-11、HCD6原則、ニールセン5属性、WCAG4原則（POUR）、スクラムの役割、ニールセンの公式 1-(1-L)^n はいずれも標準記述と整合。
2. **第5章の「ユーザー発話→ユーザー要求→システム要件」の3層変換**（予約アプリ・経費精算・AIボットの3例で反復）は教材の白眉。この構成は今後の追加問題でも活用する。
3. EC・社内システム・AIチャットボットの**3題材を全章で使い回す一貫した世界観**。実務接続と文脈定着の両方に効いている。
4. 用語集の**7フィールド構造**（一言定義/詳しい説明/関連語/混同しやすい語/実務例/試験での注意点）が114語すべてで欠落なし。
5. **A優先語句47語は用語集で完全カバー**（ビルド検証とも整合）。
6. 対比表の多用（UX/UI/CX/DX/EX、定量×定性×意識×行動の3軸表など）。第4章の3軸対比表は教材内最良の定着装置。
7. 2週間/4週間学習計画に**再テスト日（間隔をあけた復習）が既に組み込まれている**。模擬問題の章別範囲指定もチェック表と完全一致。
8. アプリは**選択肢の表示順をセッションごとにシャッフル済み**（`prepareQuestionForSession`）。模試モード（100問・100分タイマー）、間違い復習、カテゴリ別演習、カードの3状態仕分けなど基本機能は揃っている。
9. シラバス（9大カテゴリ・33小カテゴリ）と章構成・カテゴリラベルの対応構造そのものは正しい。

## 2. 品質目標（「最高品質」の定義）

修正完了時に、以下がすべて満たされている状態を目標とする。

1. **網羅**: 公式シラバス33小カテゴリの重要ワードすべてが「本文で説明あり」または「用語集で裏面あり」のどちらか（望ましくは両方）で参照できる。A優先語は本文・用語集・模試の三重カバー。
2. **正確**: 選択肢と不正解理由の不一致ゼロ。全問題で正解が一意。標準（ISO・スクラムガイド・WCAG）と矛盾する記述ゼロ。
3. **想起練習として成立**: 位置・文長・極端語・文脈外選択肢などのメタ手がかりで正答できる問題を排除。誤答選択肢は「近縁概念との混同」を突くものが標準形。
4. **弁別学習**: 混同しやすい語のペアは、必ず両方が定義され、相互参照され、少なくとも一方の問題で弁別が問われる。
5. **記憶支援**: 列挙暗記系の知識すべてに一覧表＋想起の足場（チャンク化・頭文字・数値アンカー）がある。
6. **検証可能**: 上記が `node site/build.js` の検証（既存＋§4 P4-9で追加する品質ゲート）と、チェック表の実態一致で機械的・目視的に確認できる。

## 4. 修正バックログ

各項目に: 対象・根拠（監査で確認済みの事実）・修正内容・受け入れ基準。規模は S（〜30分）/ M（〜2時間）/ L（半日以上）の目安。

### P0: 正確性・信頼性の修正（最優先・即時）

#### P0-1. 選択肢⇔不正解理由の不一致修正と全数点検 【M】
- 対象: `docs/ux_kentei_basic_hcd_study_guide.md`
- 根拠: 選択肢を書き換えた際に不正解理由が追従しなかった痕跡が**12件**確認済み。理由文が選択肢と無関係な内容（テンプレ流用）になっている。
- 確認済みの不一致（アンカー: 問番号／章とQ番号で grep）:
  - 模擬問題: **問042**-C（選択肢「速度、安さ、見た目…」に理由「インフラ指標」）、**問061**-D（理由「運用監視」が無関係）、**問064**-B（「満足した発話だけ」に「技術障害だけではない」）、**問067**-A（「機能一覧の実装順序」に「ER図など」）、**問068**-B（理由「運用監視ではない」が選択肢と無関係）、**問077**-B（「担当範囲」に「インフラ構成図」）、**問081**-D（「本番品質」に「本番接続」）、**問091**-D（「売上指標で判断」に「運用監視」）
  - 章末ミニ問題: **第5章Q2**-A（選択肢「画面項目だけを並べる」に理由「ER図などの役割です」）、**第5章Q4**-C（「タスクを細切れに並べる」に「会議運営ツールではありません」）、**第6章Q1**-A（「内部部署名だけでそろえる」に「性能改善とは別領域です」）、**第6章Q1**-D（「検索機能だけを追加」に「組織運営の話です」）
- 修正内容: 理由文を選択肢の内容に対応させて書き直す（選択肢は P1-1 で差し替える場合があるため、P1-1 対象の問は差し替えと同時に行ってよい）。その後、**全145問（模擬100＋ミニ45）について「選択肢文⇔理由文」の対応表を作って総点検**する（監査は抜き取りに近く、他にも潜在する可能性が高い）。
- 受け入れ基準: 全145問×3誤答=435組の理由がすべて対応する選択肢の内容に言及している。点検結果を作業ログとして残す。

#### P0-2. 問076の正解一意性の回復 【S】
- 根拠: 誤答扱いの選択肢B「内部データ項目は十分だが利用者の分類と合っていない」は、「システム都合の分類とユーザーのメンタルモデルの不一致」という情報設計の典型問題そのものであり、正解Dと並んで擁護可能。しかもBの理由が「容量管理。」で選択肢と無関係。
- 修正内容: 最小差分は、選択肢Bを理由文に合わせて「サーバーのストレージ容量が不足している」等の明確な非・情報設計の問題に書き換えること。
- 受け入れ基準: 4択のうち擁護可能な選択肢が正解1つのみ。

#### P0-3. 問061の実質重複選択肢の解消 【S】
- 根拠: B「すでに明確な割合だけを知りたい」とD「既に分かっている割合だけを大規模に確認したい」がほぼ同文。
- 修正内容: 一方を別の誤答（例: 定性調査に関する近縁の誤解「インタビューで市場全体の割合を推定する」等）に差し替え、理由も追従させる。

#### P0-4. シラバス網羅チェック表の過大申告の暫定修正 【S】
- 対象: guide 内 H1「# 5. シラバス網羅チェック表」（現1377行付近）
- 根拠: 本文に存在しない語を「重要語句」列に掲載済みとして表示している: **UXデザイナー**（第1章行）、**高速改善**（UXグロース行）、**ダブルダイヤモンド**（デザイン思考行）、**ユニバーサルデザイン**（アクセシビリティ行）、**PMBOK**（PjM行）、**KJ法**（定性調査行）。また第2章の「確認問題数」が小カテゴリ合計11問と読めるが実際は5問。
- 修正内容: **これらの語は P2-1 で本文に追加する予定なので、Phase 1 では第2章の問題数の数え方の修正のみ行い、語句列は Phase 3 完了後の P3-6（チェック表再生成）で実態と一致させる。** ただし P2-1 を実施しない判断をした語があれば、その語はチェック表から即時削除する。

## 7. 構造規約・制約（壊すとビルドが落ちる／表示されない）

`site/build.js` の実装調査に基づく。**コンテンツ修正前に必ず読むこと。**

### 7.1 guide のパース規約
- H1 は `# N. タイトル` の**番号**で識別（1=README, 2=試験概要, 3=本編, 5=チェック表, 8=直前復習メモ, 9=2週間計画, 10=4週間計画）。番号変更・H1追加は抽出対象にならない。**新しいH1セクションを作っても画面に出ない。**
- 章は `## 第N章 タイトル`、章内は `### N. 見出し` の番号1〜7が understand/concept/related/practice/pitfalls/miniQuiz/memo に固定対応。**セクション追加（### 8. など）は不可**。節内の構成（太字・表・段落・H2以外の追加）は自由。
- 問題ブロック書式（厳守）: ミニ=`**QN. 問題文**`、模試=`## 問NNN　[カテゴリ]`（全角スペース）→ 選択肢 `A. `〜`D. `（記号直後に半角スペース）→ `正解: **X**` → `解説: …`（**1行のみ。改行すると途切れる**）→ `不正解選択肢の理由:` → `- X: 理由`（正解以外ちょうど3つ）。
- 模試のカテゴリ名は `inferChapterNum` の正規表現で章番号に変換される。**既存33種のカテゴリラベル以外を使うとエラー**（新ラベルが必要なら build.js の正規表現も同時修正）。

### 7.2 glossary のパース規約
- `## 用語名` ＋ 日本語フィールド名の箇条書き（`- 用語:` `- 一言定義:` `- 詳しい説明:` `- 関連語:` `- 混同しやすい語:` `- 実務例:` `- 試験での注意点:` `- 重要度:` `- カテゴリ:`）。**フィールド名は完全一致必須**。

### 7.3 ビルド検証（fail 条件）
- 章数=9、各章6セクション完備、各章ミニ問題=5問、模擬問題=100問、用語集≥110語。
- 全問: ID重複なし・問題文（空白正規化後）重複なし・選択肢4つ・正解がA〜D・解説非空・不正解理由が正解以外ちょうど3つ・カテゴリ非空・章番号1〜9に解決。
- 正解位置分布: 模試=各記号15〜35問、ミニ=45問中各5〜18問。
- A優先語句は本文＋用語集＋問題文・選択肢・解説・理由のどこかに部分一致で出現必須。
- **問題総数・章数・ミニ問数を変える提案はスコープ外**（変えたい場合は build.js の定数と分布検証・チェック表・解答一覧・計画の全連動修正が必要。本方針では100/9/5を維持する前提で設計してある）。

### 7.4 md() レンダラの制約（本文で使える記法）
- 対応: 見出し・段落・箇条書き（ul/ol、**ネスト不可＝平坦化**）・表（`|---|` 区切り行必須）・**太字**・`インラインコード`・リンク。
- 非対応: **画像記法**（文字として表示されてしまう）・引用 `>`・イタリック。図解が必要な内容は表か対比リストで表現する。

### 7.5 変更してよいもの／いけないもの
- 自由: docs/*.md の本文（規約内）、docs/syllabus_keywords.json、計画・チェック表・直前メモの中身。
- 生成物: `site/js/data.js`（**編集禁止・build で再生成**）。
- アプリ変更が必要（Phase 5 でのみ触る）: `site/js/app.js`（EXAM_INFO・examFacts・CHAPTER_VISUALS・機能追加）、`site/build.js`（検証追加）、`site/index.html`（フッター確認日）。
- `main` への push で GitHub Pages に自動デプロイされる（`.github/workflows/deploy.yml` が build を実行し site/ を公開）。**push 前に必ずローカルで build 緑を確認**。

## 8. スコープ外・やらないこと

1. 章数（9）・模擬問題数（100）・ミニ問題数（各5）の変更。
2. 公式問題・推薦図書本文・他社教材の複製、合格基準や出題割合の断定記述。
3. 教材の文体変更（です・ます調、3題材の世界観、断定回避）や、§1.1に挙げた強みのリファクタ。
4. アプリの大規模刷新（フレームワーク導入等）。P4はバニラJSの延長で行う。
5. 非公式情報（体験記）を公式情報と混同させる記述。使う場合は必ず出所と非公式である旨を明示。
